# 安全策略

## 支持范围

安全修复只面向仓库中最新的两个开源 HTML 文件。旧版本或其他人重新打包的副本不保证获得修复。

## 报告安全问题

如果仓库启用了 GitHub Private Vulnerability Reporting，请优先使用该功能。否则请通过仓库所有者公开资料中提供的私下联系方式报告，不要在公开 Issue 中粘贴漏洞利用代码、Token、内部 URL 或含敏感内容的文档。

报告建议包含：

- 受影响的文件名和版本。
- 浏览器及操作系统版本。
- 最小复现步骤。
- 实际影响和建议修复方向。
- 已脱敏的截图或示例文件。

仓库建立后，维护者应在此补充明确的安全联系渠道和预计响应时间。

## 安全边界

- 本地 Markdown 默认在浏览器中处理。
- 页面运行时会从公共 CDN 加载第三方脚本、样式和字体。
- 从 URL 打开文档、加载远程图片或启用 CORS proxy 会访问外部网络。
- 只有用户主动配置并执行 Share 时，文档才会发送到上传服务。
- 分享服务请求头可能包含 Token，并以明文保存在当前浏览器的 `localStorage`。
- 导出的 HTML 和 Word 文档不应包含分享服务配置。

## 使用者建议

- 只从可信发布来源下载 HTML，并校验发布者提供的哈希。
- 不要在公共设备中保存包含 Token 的分享配置。
- 分享配置前检查 `Copy JSON` 内容并移除敏感请求头。
- 企业环境应自行托管依赖和上传服务，并配置最小权限、CORS 和内容类型校验。
- 不要把不可信的上传地址或文件库地址配置到工具中。

## 维护者建议

- 固定第三方依赖版本并定期检查安全公告。
- 保留 DOMPurify 清理步骤，谨慎评估任何放宽 HTML 或 Mermaid 安全模式的改动。
- 发布前执行 `docs/RELEASE_CHECKLIST.md` 中的敏感信息和导出泄漏检查。
- 不要在 Issue、测试夹具、截图或提交历史中保存真实公司凭据。
